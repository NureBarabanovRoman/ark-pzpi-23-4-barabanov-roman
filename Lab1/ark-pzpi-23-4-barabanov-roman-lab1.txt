3 ХІД РОБОТИ
     Процес створення документа Vision & Scope розпочався з етапу збору вихідної інформації про майбутню систему. Спочатку було визначено основну ідею проекту, його мету та потенційних користувачів. На цьому етапі проводився аналіз проблем, які має вирішувати система, а також дослідження можливих сценаріїв її використання. Це дозволило чітко окреслити контекст розробки та зрозуміти, яку цінність продукт приноситиме своїй аудиторії.
Далі було проведено огляд аналогічних рішень, що вже існують на ринку. Таке порівняння допомогло виявити недоліки конкурентів і визначити можливості для покращення, які зможе реалізувати проект. Після цього сформувалася концепція системи – набір ключових ідей, що визначають її функціональність, структуру та відмінні риси.
На наступному етапі відбувалася деталізація вимог і узгодження основних компонентів системи. У документі було описано загальну архітектуру, межі проєкту, очікувані результати та ключові критерії успіху. Для кожного модуля визначено головні завдання, способи взаємодії з іншими частинами системи та рівень пріоритетності.
Завершальним етапом стало узагальнення отриманих результатів у єдину структуру документа. В Vision & Scope були зведені всі ключові аспекти: від концепції та цілей до технічних вимог і перспектив розвитку системи. Це забезпечило чітке розуміння напрямку розробки як для команди, так і для зацікавлених сторін.



4 ВИСНОВКИ

Документ став базовим орієнтиром для подальших етапів проектування та розробки. Він слугує фундаментом для планування реалізації, розподілу завдань і визначення пріоритетів розвитку системи.

ДОДАТОК Б 
Vision & Scope
1.1 Аналіз проблеми 
Традиційні методи збору думок (анкети, QR-коди) вимагають від користувача значних зусиль, що знижує кількість відповідей у публічних місцях. Крім того, існуючі системи часто не надають аналітики в реальному часі та не дозволяють інтегрувати фізичні пристрої для голосування разом з онлайн-формами. Система покликана вирішити цю проблему шляхом створення гібридної платформи, де голоси збираються через зручні фізичні IoT-термінали та мобільні додатки, а результати обробляються миттєво.
1.2 Аналіз аналогів 
Існуючі сервіси: SurveyMonkey, Google Forms, Mentimeter – орієнтовані переважно на онлайн-користувачів. Спеціалізовані термінали зворотного зв'язку часто є дорогими пропрієтарними рішеннями. Недоліки аналогів:
* Низька залученість офлайн-аудиторії.
* Висока вартість апаратних рішень.
* Обмежені можливості математичного аналізу даних у базових версіях.
Нова система дозволить використовувати доступні IoT-компоненти для збору голосів та надаватиме розширену статистику через єдиний API.
2.1 Опис призначення системи 
Система призначена для проведення швидких експрес-опитувань та збору зворотного зв'язку. Користувач може проголосувати натисканням фізичної кнопки або через смартфон, а адміністратор – миттєво побачити результати. Цінність:
* Зручність та швидкість голосування.
* Гібридний формат збору даних (онлайн + офлайн).
* Автоматичний розрахунок достовірності даних (аналітика).
2.2 Опис бізнес-логіки
1. Адміністратор створює опитування через веб-інтерфейс.
2. IoT-пристрій або мобільний клієнт фіксує вибір користувача.
3. Дані надсилаються на сервер (HTTP-запит) разом з ідентифікатором пристрою.
4. Сервер зберігає голос, перевіряє валідність пристрою та перераховує статистику.
5. Користувач або адміністратор переглядають оновлені графіки та аналітику.
3.1 Опис функціональності серверної частини та IoT-клієнта
 Серверна частина:
* Прийом голосів та реєстрація пристроїв.
* Збереження даних у реляційній базі даних.
* Розрахунок статистики (ентропія, статистична похибка).
* API для взаємодії з клієнтами.
IoT-клієнт:
* Фіксує натискання фізичних кнопок вибору варіанту.
* Надсилає дані на сервер через HTTP POST запит.
* Періодично надсилає статус "серцебиття" (заряд батареї, статус мережі).
3.2 Опис функціональності веб-клієнта та мобільного клієнта
 Веб-клієнт:
* Адмін-панель для створення та редагування опитувань.
* Візуалізація результатів (діаграми, графіки).
* Управління списком підключених пристроїв.
Мобільний клієнт:
* Інтерфейс для голосування користувачів.
* Перегляд публічних результатів.
* Отримання повідомлень про нові опитування.
Взаємодія із сервером:
* Здійснюється через REST API з використанням формату JSON.
4.3 Опис програмних засобів та інструментів
 Проект складається з кількох основних частин:
* Серверна частина реалізована мовою Python з використанням фреймворку FastAPI, що забезпечує високу продуктивність. Валідація даних здійснюється бібліотекою Pydantic, а документація генерується автоматично через Swagger UI/OpenAPI.
* Робота з даними: використовується об'єктно-реляційна система управління базами даних PostgreSQL. Взаємодія з БД реалізована через ORM SQLAlchemy та високошвидкісний асинхронний драйвер asyncpg.
* IoT-пристрій: емулюється або реалізується на базі мікроконтролера ESP32, програмується мовою MicroPython. Обмін даними відбувається через протокол HTTP.
* Клієнтська частина: Веб-клієнт створюється з використанням HTML/JS, мобільний клієнт адаптується під сучасні платформи.
* Інструменти: GitHub для контролю версій, PyCharm для розробки бекенду та Wokwi для симуляції IoT.
